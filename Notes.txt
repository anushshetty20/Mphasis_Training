Granularity:
1.Single table strategy -> all classes in hierarchy are mapped to a single table
		problem -> multiple null values
2.Joined table strategy -> base class contains common attribute and sub class contain subclass specific attribute with foreign key linking them
		problem -> it need complex joins and multiple joins
3.Table per class Strategy -> each class in the hierarchy has its own table with all attribute from the class and its superclass
		problem -> data redundancy as common attribute are repeated across multiple tables

N+1 problem in Hibernate:
	executes one initial database query to retrieve list of parent entries and then N additional queries to fetch lazily loaded child entries

Hibernate Objects : 

1. Transient -> exist in heap as normal java objects. Hibernate doesn't manage transient entities or persist changes done on them.

			POJO
			 |
			 |
		       CLIENT    [Transient objects are independent of Hibernate]


2. Persistent -> exist in DB -> Hibernate manages the persistence for Persistent Object

			POJO <--> session <--> Database <--> POJO table
			 ||
			 ||
		       CLIENT
		[Persistent objects exist in the database and Maintained by Hibernate]
	[if fields are changed on a persistent object, Hibernate will keep the DB representation up to date when application marks the changes as to be committed]


3. Detached : entities have representation in DB, but changes to entity will not be reflected in the DB and vice-versa.
		
		Detached Object
			POJO                Database <--> POJO table
			 ||								 
			 ||
		       CLIENT
	[Detached objects exist in the database but are not maintained by Hibernate]
